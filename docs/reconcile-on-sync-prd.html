Reconcile POPS↔Local Conflicts

Let ops resolve duplicate / mis-named / orphaned rider records self-service and safely — without turning the rider sync button into a destructive operation.

If you read nothing else

★Summary

Conflicts between POPS and the local DB (a rider deleted in POPS, a duplicate rider_id, a record whose name drifted from POPS) should be resolvable by ops without an engineer. But the resolution is destructive — it reassigns route/reimbursement history and deletes records — so it must stay out of the routine sync button and behind an explicit, previewed confirmation.

1
Phase 1 — sync surfaces conflicts  DO FIRST
Keep sync a safe, idempotent upsert, but return a structured conflicts[] instead of swallowing them as a silent failed count. Reuses detection already written in purge_legacy_users. ~0.5 day, non-destructive.
2
Phase 2 — guided reconcile  IF RECURRING
An admin review queue that, per conflict, shows a dry-run preview of a recommended resolution ("would reassign 244 shipments, delete 2 shells, rename X→Y") and applies it transactionally via the command's vetted logic. ~3–5 days.
Why not auto-clean on sync? Sync is a frequently-clicked, currently-safe operation. Making it delete/merge records erodes the guardrails (dry-run-first, transaction, reassign→verify→delete, "refuse on incomplete pull"). And POPS-origin duplicates must be fixed in POPS, not papered over locally — auto-resolving hides upstream problems that recur every sync.

1The problem (from the real cleanup)

Today sync_riders_from_pops (apps/authentication/views.py) is an upsert: it walks the riders POPS returns and updates-or-creates each. It has created/updated/failed counters and no prune step. Observed consequences:

ConflictWhat happens todayReal example
Orphaned localRider deleted in POPS → local row lingers; sync ignores it forever.Deleting pops 70 left Sheshagiri_300099 with 214 shipments dangling.
Rename collisionPOPS rider_id ≠ local for the same pops_rider_id, and the target name is taken by a stale row → IntegrityError → silent failed++. Sync can never fix it.30009 (pops 92) couldn't become Seshagiri_300099 while a dead record held that name.
POPS-origin duplicateSame rider_id under several pops ids. Not a local bug — can't be deduped locally.11718 ×3 (pops 87/88/89), Others ×5.
Why this is dangerous to automate naively. History is string-keyed by employee_id (= username / rider_id), not a foreign key — deleting a record orphans its route/shipment/ack history. A "just pick which to keep and merge" wizard would: (a) happily merge rider history into a manager account (a near-miss this session — 12180 rider-test data nearly landed on the Google-SSO admin); and (b) under-state blast radius if it only counts route GPS — one record looked like "0 history" but actually carried 244 shipments + 170 acknowledgments.

2Phase 1 — sync surfaces conflicts DO FIRST

Small, additive, non-destructive. Ends the "silent failed" blind spot.

Change sync_riders_from_pops to return, alongside created/updated/failed, a structured conflicts[]:

  • orphan_local — local pops_rider_id not in the pull (deleted in POPS).
  • rename_collision — POPS rider_id ≠ local for a shared pops_rider_id, and the target name is already taken (the case that currently silently fails).
  • rider_id_mismatch — POPS rider_id ≠ local, target free (sync could rename — flag for visibility).
  • pops_dup — same rider_id under multiple pops ids → label "fix in POPS".

Detection already exists in purge_legacy_users (_report_rider_mismatches + the POPS-dup scan). Extract a shared reconcile_detect(pops, local) helper both call — no new logic, just surfacing.

Outcome: the sync result (and the admin UI) shows conflicts explicitly instead of a meaningless "1 failed". Nothing is deleted. Ops can then act (today: hand to an engineer + the command; after Phase 2: self-serve).

3Phase 2 — guided reconcile IF RECURRING

Human-in-the-loop, explicit, previewed — the destructive part isolated from sync.

Admin "Reconcile" queue

Mirror the existing pending-approvals / VehicleChangeRequests panels. Lists Phase-1 conflicts, grouped by person where inferable. Each row shows a recommended resolution (pre-selected) and lets the user override (pick the survivor).

Per-conflict dry-run preview

POST /auth/reconcile/preview runs the cleanup logic in dry-run for the proposed resolution and returns the consequences as JSON — "would reassign 244 shipments + 170 acks, rename 30009→Seshagiri_300099, delete 2 shells". No silent auto-remap; the operator sees the true blast radius before confirming.

Apply

POST /auth/reconcile/apply — transactional, reusing the command's vetted reassign→verify-zero-links→delete/rename path.

Prerequisite: factor purge_legacy_users into a shared service module so the management command and the endpoint share one tested code path. No logic duplication — the command becomes a thin CLI over the service; the endpoint a thin HTTP wrapper.

4Safety invariants — non-negotiable; every one learned the hard way

  • No cross-role merge. Rider history must never merge into a manager/staff account; resolutions are rider↔rider / staff↔staff only. (The 12180 near-miss.)
  • Never act on an incomplete POPS pull. Gate on pull-ok + a minimum rider count; "not in POPS" off a truncated pull would wrongly delete real riders.
  • Reassign → verify zero links → only then delete. Never orphan history; prefer soft-archive for riders when unsure.
  • Size a merge by ALL string refs — Shipment.employee_id, acknowledgments, events — not just route GPS. (A "0 history" record was really 244 shipments.)
  • POPS-origin duplicates are flagged, not auto-merged. Fixed at the source; surface "fix in POPS." (Others, 11718.)
  • A human confirms a previewed resolution. No silent auto-remap on a routine action.

6Recommendation

Ship Phase 1 now — cheap, safe, and it ends the silent-failure problem (conflicts become visible instead of vanishing as failed).
Build Phase 2 only if conflicts recur. Today's volume is low — the legacy cruft cleared in the 2026-06-24 pass plus one Seshagiri consolidation. When it does recur, the service-refactor step is what keeps command + endpoint on one tested path.

Until Phase 2 exists, conflicts are resolved with the purge_legacy_users management command (dry-run default; --map, --rename, --delete-orphan-shadow, --archive-rider, --delete-history-for) — see CLAUDE.md.
