★Summary

RiderPro already tracks every rider's GPS and computes road-aware routes & ETAs. Two high-value steps build on that — neither needs new infrastructure or a rebuild:

1
Automatic mileage & fuel cost  MOSTLY BUILT
Stop manually entering each rider's start/end km. Distance comes from the GPS trail; fuel litres & petrol cost come from the rider's vehicle mileage & current fuel price — automatically.
2
Route planning with an ops approval gate  PROPOSED
RiderPro auto-routes each rider (pickups + deliveries); ops sees overlaps & can reassign before approving & dispatching; riders execute with ETAs. Reuses the existing optimizer.
What we deliberately skip: auto-assigning shipments to drivers, zones, multi-driver batch solving, background queues, advanced solvers. The order system already assigns each shipment to a rider — so none of that is needed today.

The ask: confirm priority (mileage first?), and answer the 6 open questions (last section) to finalise the planning spec.

2The problems we're solving

⛽ Manual mileage entry

Today someone keys in each rider's start/end km by hand to work out mileage and petrol cost — slow, error-prone, and hard to audit.

👁️ No pre-shift visibility

Ops can't see the day's planned routes, total distance, or finish times before drivers leave the depot.

⚖️ No balance check

An overloaded rider, or a shipment with no rider / no map location, is only discovered mid-shift.

📞 No early ETAs / no locked plan

Routes re-order live as drivers move; ops can't answer "when will it arrive?" until they're already out.

✅ Already in place

  • Per-stop pickup/delivery confirmations (the rider taps each)
  • Road-distance routing (OpenRouteService) between any points
  • Per-rider vehicle & mileage, plus fuel pricing, already modelled
  • Fields to store distance, fuel litres & cost on every session

🔧 What this adds

  • Distance = road distance between confirmed stops (start → each pickup/delivery → end)
  • Auto fuel cost = km ÷ the rider's vehicle mileage × current fuel price
  • Surface per rider in analytics → manual km entry goes away
  • GPS trail kept only as a fallback + sanity check
⚠️ Why stop-to-stop, not the live GPS trail: RiderPro is a web app — GPS only records while it's open and in the foreground. If a rider locks their phone or closes the browser mid-route, the trail has gaps and under-counts distance. The confirmed pickup/delivery taps are reliable, so road distance between them gives fair, gap-proof mileage. (True background tracking would need a native app — see Web vs native app.)

Three pillars

🧭
Auto-route + mileage

System orders each rider's stops (pickups + deliveries) by real travel time, with ETAs and estimated km & fuel.

🔀
Ops review & smart reassign

Flags overlaps (two riders covering the same area) and suggests moving an order to a nearer rider. Ops decides, then approves & dispatches.

🛵
Rider executes

Gets the locked, ordered route + ETAs; confirms each pickup/delivery; actual mileage & fuel auto-recorded on completion.

Expected benefit: optimized ordering on real road distances typically cuts ~10–25% of distance (and fuel) vs ad-hoc ordering — and the day-view makes your actual saving measurable.

ETAs are realistic & live: each stop's ETA = driving time plus a configurable service-time allowance per stop (handover, POD signature) — not just driving time. As the rider completes stops and moves, the route re-runs so arrival clock-times shift live — running 15 min behind at stop 3 pushes stops 4+ automatically. Ops & customers see real arrival windows, not optimistic ones.

3What we deliberately won't build

A textbook delivery-routing system assumes the software decides assignments. Ours doesn't — the upstream system already does. So most of that machinery is dead weight.

Batch multi-driver optimization ("split 80 stops across 5 drivers")NoEach stop already belongs to a rider — no pool to split.

Implication: this collapses from "build a routing engine" to "add a thin mileage calc + a planning & oversight layer on top of what already works." Far less code, far less risk.

  • IN Problem flags: overload, unassigned, unmappable
  • IN Overlap detection + reassignment suggestions; ops approves & dispatches
  • IN Rider gets the locked ordered route + ETAs; confirms pickup/delivery per stop

🚫 Out of scope (and why)

  • OUT Auto-assignment / cross-driver balancing — decided upstream
  • OUT Zone-based assignment — decided upstream
  • OUT Batch multi-driver solving — nothing to split
  • OUT Background job queue — no heavy batch
  • LATER Advanced solver — only with time-windows/capacity

Where the value is (impact vs effort)

Auto-mileage and the ops day-view are high-impact / low-effort; in-app rebalancing is polish.

8Success metrics

Operational

  • Manual km entries eliminated; fuel cost auto-computed per rider
  • Ops sees 100% of planned routes + finish ETAs before dispatch
  • Overloaded / unassigned / unmappable shipments flagged before shift
  • Lower km & fuel per route (ordered vs ad-hoc); fewer "where's my driver?" calls

Leading indicators

  • % of riders with auto-computed (vs manual) mileage
  • % of days dispatched via the planner; % routes locked before first delivery
  • Auto-km vs odometer gap (validates the mileage calc)
  • Planned-vs-actual finish gap (calibrates the ETA estimate)

9Future option — a smarter solver, when it's worth it

We don't have delivery time-windows today, so an advanced constraint solver (such as Google's OR-Tools) is explicitly out of scope for now. This records why and when it becomes worth it.

Adopt a smarter solver when any of these become real:
  • Delivery time-windows — "deliver only 10:00–12:00." Today's simple ordering can't honour windows; a solver optimises within them. (Flagged as a likely future need.)
  • Pickup-before-delivery precedence — if a picked-up item must be collected before its own delivery stop. Since routes already mix pickups & deliveries, this is the most likely first trigger. (Simple ordering handles independent pickups/deliveries fine; only same-item precedence needs a solver.)
  • Vehicle capacity — bike vs van load limits per route.
  • Assignment moves into RiderPro — if we ever stop receiving pre-assigned riders and must split a pool across drivers.

Until then, today's road-aware ordering is sufficient, and the routing layer is already swappable — so changing the ordering strategy later is contained and low-risk.

11Web app vs native app

The web app can't track GPS in the background (it only records while open & in the foreground). That's the main reason "should we build an app?" comes up. Here's the honest trade-off.

✅ What a native app gains

  • True background GPS — keeps tracking with the screen off / app backgrounded (the web app cannot). High-fidelity live map + a continuous trail for disputes.
  • Push notifications — instant new-assignment / reassignment alerts to riders.
  • Better offline & battery-managed native location services.
  • App-store presence; smoother, more "app-like" feel.

💸 What it costs / loses

  • Build + ongoing maintenance of a native (or hybrid) app alongside the web app.
  • App-store overhead — Apple $99/yr + Google $25; review delays on every release (no instant hot-fix like a web deploy).
  • "Always-allow location" permission — app-store privacy justification; some riders may decline.
  • Battery drain from continuous background GPS on rider phones.
Recommended middle path — Capacitor wrapper: wrap the existing React web app in a thin native shell (Capacitor) and add a background-geolocation plugin. You get true background tracking without rewriting the app, and keep deploying the web code as today. Far cheaper than a full native rebuild.
My recommendation: stay web for now. The stop-to-stop mileage already removes the biggest reason to rush native (fair mileage without needing a continuous trail). Move to a Capacitor wrapper when you genuinely need true live background tracking — e.g. a high-fidelity real-time ops map, or a continuous breadcrumb trail for delivery disputes.
