★Summary

RiderPro tracks GPS and computes road-aware routes & ETAs in production. As of this update, automatic mileage & fuel cost is also live. The remaining step is the planning feature:

1
Automatic mileage & fuel cost  LIVE
Replaces manual start/end-km entry. Distance = road distance between the rider's confirmed stops (robust to web-app GPS gaps); fuel litres & cost from the rider's vehicle mileage & current fuel price — automatic. Calibrate against your manual numbers before trusting it for reimbursement.
2
Route planning with an ops approval gate  PROPOSED
RiderPro auto-routes each rider (pickups + deliveries); ops sees overlaps & can reassign before approving & dispatching; riders execute with live ETAs. Reuses the existing optimizer.
✅ Live in production (deployed & verified): road-aware routing · auto stop-to-stop mileage & fuel cost (now using each rider's actual vehicle mileage) · service-time + live ETAs with arrival clock-times · debounced re-optimization · stop "please wait" overlay · vehicle control & approval (day-start confirmation + rider request → admin approve/reject).
📋 Proposed — not built: the rider-driven planning feature (ops day-view, order/lock/dispatch, overlap detection + reassignment). We deliberately skip auto-assignment, zones, batch solving, queues & advanced solvers — the order system (PIA) already assigns each shipment to a rider.

The ask: calibrate the live mileage/ETA against your manual numbers, then answer the open questions to green-light the planning feature.

2The problems we're solving

⛽ Manual mileage entry

Today someone keys in each rider's start/end km by hand to work out mileage and petrol cost — slow, error-prone, and hard to audit.

👁️ No pre-shift visibility

Ops can't see the day's planned routes, total distance, or finish times before drivers leave the depot.

⚖️ No balance check

An overloaded rider, or a shipment with no rider / no map location, is only discovered mid-shift.

📞 No early ETAs / no locked plan

Routes re-order live as drivers move; ops can't answer "when will it arrive?" until they're already out.

  • Per-stop pickup/delivery confirmations (the rider taps each)
  • Road-distance routing (OpenRouteService) between any points
  • Per-rider vehicle & mileage, plus fuel pricing (modelled)

  • Each session snapshots the mileage used (immutable reimbursement history)

Only operational step left for mileage: calibrate the auto figures against your manual numbers before reimbursement use. The remaining build work is all in ② route planning.

⚠️ Why stop-to-stop, not the live GPS trail: RiderPro is a web app — GPS only records while it's open and in the foreground. If a rider locks their phone or closes the browser mid-route, the trail has gaps and under-counts distance. The confirmed pickup/delivery taps are reliable, so road distance between them gives fair, gap-proof mileage. (True background tracking would need a native app — see Web vs native app.)

✅ How vehicle control works (live)

Mileage drives the fuel-cost money, so the vehicle that sets it is governed, not freely rider-editable. This is now live, reusing the existing signup-approval pattern:

1 · Admin owns it

Admin assigns/approves each rider's vehicle type — the official mileage source (RiderAccount.vehicle_type).

2 · Rider proposes

A rider can add/change their vehicle, but it's a request → admin approval (same flow as signup approval). Captures reality without letting the money input be gamed.

3 · Confirm at day-start

At session start the rider confirms "Today's vehicle: Bike · 45 km/l — Confirm / Request change," or (no vehicle) starts anyway with a flag. Defaults to their approved vehicle.

✅ Shipped: admin-approved vehicle + rider-proposed changes (admin approve/reject) + day-start confirmation, plus a per-session mileage snapshot so past fuel costs don't change if the vehicle is later updated. Pure rider self-select (no approval) was deliberately avoided — it would let a reimbursement input be gamed.

Expected benefit: optimized ordering on real road distances typically cuts ~10–25% of distance (and fuel) vs ad-hoc ordering — and the day-view makes your actual saving measurable.

✅ Live — ETAs are realistic & live: each stop's ETA = driving time plus a configurable service-time allowance per stop (handover, POD signature) — not just driving time. As the rider completes stops and moves, the route re-runs (debounced to protect the routing quota) so arrival clock-times shift live — running 15 min behind at stop 3 pushes stops 4+ automatically. (The optimizer & ETAs are live; what's still proposed is the ops-facing planning screen around them.)

3What we deliberately won't build

8Success metrics

Operational

  • Manual km entries eliminated; fuel cost auto-computed per rider
  • Ops sees 100% of planned routes + finish ETAs before dispatch
  • Overloaded / unassigned / unmappable shipments flagged before shift
  • Lower km & fuel per route (ordered vs ad-hoc); fewer "where's my driver?" calls

Leading indicators

  • % of riders with auto-computed (vs manual) mileage
  • % of days dispatched via the planner; % routes locked before first delivery
  • Auto-km vs a one-time manual calibration sample (validates the mileage calc)
  • Planned-vs-actual finish gap (calibrates the ETA estimate)

9Future option — a smarter solver, when it's worth it

We don't have delivery time-windows today, so an advanced constraint solver (such as Google's OR-Tools) is explicitly out of scope for now. This records why and when it becomes worth it.

Adopt a smarter solver when any of these become real:
  • Delivery time-windows — "deliver only 10:00–12:00." Today's simple ordering can't honour windows; a solver optimises within them. (Flagged as a likely future need.)
  • Pickup-before-delivery precedence — if a picked-up item must be collected before its own delivery stop. Since routes already mix pickups & deliveries, this is the most likely first trigger. (Simple ordering handles independent pickups/deliveries fine; only same-item precedence needs a solver.)
  • Vehicle capacity — bike vs van load limits per route.
  • Assignment moves into RiderPro — if we ever stop receiving pre-assigned riders and must split a pool across drivers.

Until then, today's road-aware ordering is sufficient, and the routing layer is already swappable — so changing the ordering strategy later is contained and low-risk.

11Web app vs native app

The web app can't track GPS in the background (it only records while open & in the foreground). That's the main reason "should we build an app?" comes up. Here's the honest trade-off.

The core difference, side by side

Dimension🌐 Web app (today)📱 Native / hybrid app
Background GPS (screen off / backgrounded)❌ No — only while open & foregrounded✅ Continuous
Location accuracy⚠️ Browser-grade✅ Native, finer
Push notifications (new/reassigned orders)⚠️ Limited / unreliable✅ Reliable
Ship a fix / update✅ Instant (web deploy)❌ App-store review delay
Getting it on a rider's phone✅ Just a URL (or PWA install)⚠️ App-store download
Build & maintenance cost✅ One codebase, already built❌ Extra build + store fees + upkeep
Battery use✅ Lighter⚠️ Heavier (background GPS)
Offline resilience⚠️ OK (IndexedDB queue)✅ Stronger

Pros & cons of each

🌐 Web app (current)

Pros

  • Nothing to install — open a URL; instant updates via web deploy
  • One codebase, already built & live; lowest cost
  • Lighter on battery; no app-store gatekeeping or review delays

Cons

  • No background GPS — tracking pauses when the app is closed/backgrounded
  • Weaker push notifications; browser-grade location accuracy
  • Can't physically stop a rider closing the tab mid-route

📱 Native / hybrid app

Pros

  • True background GPS — continuous trail + high-fidelity live map
  • Reliable push notifications; better location accuracy & offline
  • App-store presence; smoother, more "app-like" feel

Cons

  • Extra build + ongoing maintenance alongside the web app
  • App-store fees (Apple $99/yr, Google $25) + review delays on every release
  • "Always-allow location" permission friction; heavier battery use
Bottom line: stick with the web app — the only thing a native app truly buys us is background GPS tracking (recording while the rider's phone is locked or in a pocket), and we don't need that today because mileage is measured from the rider's confirmed stops, not a continuous live trail.
Recommended middle path — Capacitor wrapper: wrap the existing React web app in a thin native shell (Capacitor) + a background-geolocation plugin — true background tracking without rewriting the app, still deploying the web code as today.

In plain business terms: it's like fitting your existing delivery van with an always-on GPS tracker instead of buying a brand-new van. You keep everything you've already built — same team, same codebase, same instant fixes — and pay only for the one part you're missing (always-on location), at a fraction of the cost and time of a full native rebuild.
My recommendation: stay web for now. The stop-to-stop mileage already removes the biggest reason to rush native (fair mileage without needing a continuous trail). Move to a Capacitor wrapper when you genuinely need true live background tracking — e.g. a high-fidelity real-time ops map, or a continuous breadcrumb trail for delivery disputes.
